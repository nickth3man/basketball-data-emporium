# Initialization Plan: Basketball Data Emporium API ← DuckDB → Next.js UI

> **Status:** Phase 1 complete (sidecar skeleton + `/api/status` live). This plan covers the remaining initialization work: correctness scaffolding (Track A), the 14 remaining API endpoints (Track B, Phases 2–9), and frontend type reconciliation (Track C).
>
> **Sources synthesized:** explorer schema recon, librarian domain/framework research, oracle connection plan + verification strategy, librarian golden dataset, fixer Phase 1 build report.

## Overview

The `basketball-data-emporium` repo contains a **22 GB DuckDB file** (`data/nba.duckdb`, v1.4.2, `unified_star` conformed schema + 21 `api.v_*` projection views) and a fully-wired **Next.js 16 frontend** that was built to consume a FastAPI sidecar (`basketball-data-emporium`) at `http://127.0.0.1:8765`. **That sidecar did not exist in the repo** — the frontend's hardened HTTP client (`frontend/src/lib/api-client.ts`), 1596-line generated type contract (`frontend/src/lib/openapi-types.ts`), 8-code error envelope, and React Query hooks all pointed at a missing service.

**The goal of initialization:** rebuild the missing FastAPI sidecar so the frontend lights up against the real DuckDB, with a correctness scaffolding that proves the right data flows to the right UI surfaces — then implement the remaining 14 endpoints in dependency order, culminating in a fully-functional player/team hub.

**Phase 1 is done.** This document plans Phases 2–9 plus the correctness layer that gates them.

## Architecture Decisions

### Decision 1 — Rebuild the FastAPI sidecar (Option A), not Next.js Route Handlers (Option B)

**Rationale:** the frontend was designed to talk to a FastAPI service whose `/openapi.json` drives the entire type pipeline. That design is load-bearing in five places we would have to *emulate* under Option B:

1. FastAPI's `HTTPValidationError` 422 shape (`openapi-types.ts:417-421, 604-616`)
2. The `{ detail: { code, message, detail } }` envelope (`api-errors.ts:82-95`)
3. Auto-generated operation IDs
4. The `gen:api` codegen loop (`package.json:11` → `scripts/generate-api-types.ts`)
5. The `rate_limit_jailed` server-side jail that implies shared cross-request state — impossible in a stateless Route Handler.

Three clinchers: (a) `rate_limit_jailed` **cannot** be implemented honestly in stateless Route Handlers; (b) `openapi-types.ts` carries the header `auto-generated by openapi-typescript — Do not make direct changes`, and only a live FastAPI `/openapi.json` keeps it auto-generated; (c) the two-process topology is already paid for — Playwright is configured for UI:3000 + API:8765 (`frontend/playwright.config.ts`).

### Decision 2 — DuckDB access: read-only, pooled, no ORM

- **Always read-only** (`duckdb.connect(path, read_only=True)`). The API never writes. DuckDB is exclusive-writer; read-only allows concurrent readers.
- **Pool of 6 connections** (DuckDB is one-query-per-connection). Module-level singleton via double-checked locking in `basketball_data_emporium/db/pool.py` so `--reload` doesn't reopen the 22 GB file per request.
- **No ORM.** Drizzle DuckDB dialect is maintainer-labeled "Experimental" (~20 stars); Prisma has no DuckDB connector. Raw parameterized SQL via `duckdb` prepared statements is the canonical OLAP pattern.

### Decision 3 — Query routing by schema layer

| Schema layer        | Role                                  | When to query                                                |
| ------------------- | ------------------------------------- | ------------------------------------------------------------ |
| `api.v_*` (21 views) | Projection layer the original service used | Dataset endpoints (#6, #7, #13, #14) — encapsulate joins     |
| `unified_star.*`    | Clean conformed star (20 tables)      | Cheap lookups (#3, #4, #10, #11, summary headers #5, #12)    |
| `meta.*`            | Column catalog (345 canonical metrics, 12,678 mappings) | Hydrate `ColumnMeta[]` labels in Phase 5+                    |
| `audit.*`           | Data-quality layer (31 tables)        | **NOT deferred** — first-class correctness asset (see Decision 5) |
| `xref.*`, `nbadb.*`, `main.*` | Identity resolution / SQLite mirror / legacy star | DEFERRED (YAGNI)                                             |

### Decision 4 — Honor the existing `openapi-types.ts` contract unchanged

The 1596-line committed file documents all 15 endpoints with hand-curated descriptions and a custom operationId generator. **Phase 2 must resolve a generator mismatch** (fix-1 flagged this) before regenerating the full file — otherwise the drift gate reports false positives.

### Decision 5 — `audit.*` is a first-class correctness asset, not deferrable

The prior deferral of `audit.*` was wrong for *correctness*. `audit.pipeline_run_log`, `audit.dq_results`, `audit.metric_discrepancy`, `audit.cross_table_discrepancy` are the answer to "did the last ETL pass?" and must gate `/api/status` from Phase 2 onward.

### Decision 6 — Local BBR anchor HTML is the primary ground truth where it exists

`data/anchors/bbref-pages/` contains **423 HTML files (251 MB)** — direct grabs from basketball-reference.com, snapshotted **2026-06-14** against BBR's `req/202605271` build (with a 9-file re-fetch on 2026-06-25). Structure:

- 378 unique by MD5 (45 byte-identical dupes from BBR's tabbed page structure), 271 unique canonical URLs.
- Organized by BBR page-type subdirs: `player_career_stats/`, `player_splits/`, `player_shot_charts/`, `team_roster/`, `team_box_scores/`, `play_by_play/`, `players_season_totals/`, `players_advanced_season_totals/`, `franchise_*/`, `draft/`, `allstar/`, `season_schedule/`, etc.
- **No manifest.** No `index.json`/`README`/`*.csv` sibling — just raw HTML. Retrieval date is reconstructable only from file mtimes + the BBR `<meta name="revised">` per-fetch stamp.

**Why this matters for correctness:** local anchors are **immutable ground truth**, immune to BBR bot-mitigation, republishing, and the "wrong but stable" risk of live-URL pinning. Where an anchor exists for a golden fact, it is the **primary source**; the live URL is the fallback. Where no anchor exists, the golden value stays pinned to the live URL (with the annual-refresh caveat) until an anchor is fetched.

**Coverage of the 37-fact golden dataset (see Golden Dataset Reference for the per-fact matrix):**
- **10 facts covered** (8 full + 2 partial): player careers for `jamesle01`/`curryst01`/`jordami01`/`abdulka01`/`chambwi01`/`bryanko01`/`duranke01`; `team_roster/BOS_2024.html`; `players_advanced_season_totals/2024_false.html`; `attendance/2024.html` (NBA_2024 season summary).
- **27 facts have no local anchor** and stay on live URLs: most player-seasons (Curry 2015-16, Jordan 1986-87, Wilt 1961-62, both Harden trade splits, Tatum 2023-24), most team-seasons (CHI 1995-96, GSW 2015-16, LAL 1971-72), franchise leaders (LAL/BOS PTS), both game boxscores (Kobe 81, Wilt 100), most 2023-24 league leaders.
- **Large surplus** of anchors with no golden fact: 11 ATL franchise pages, ~75 league pages for non-2024 seasons, Bird/Duncan/Russell/Paul career pages, 8 All-Star pages, 15 draft pages, coach/exec/referee pages, 70+ friv pages. These are candidate new golden facts or future anchor-fetch targets.

**Operational rule:** `golden.csv` carries a `source_type` column (`anchor_html` | `live_url`) and an `anchor_path` column (relative path under `bbref-pages/`). The harness resolves the anchor first; only facts with `source_type=live_url` hit the network. **Task A0 fetches the ~18 missing anchor pages before A2 pins `golden.csv`** so the golden set achieves maximum immutable coverage from day one. A follow-up task (A5) generates the missing manifest so the corpus is self-describing.

**Curated surplus promotion:** the corpus contains high-value anchored pages with no golden fact (Bird/Duncan/Russell/Paul careers, ATL franchise leaders, 2026 Finals boxscore, BOS 1979-80 team season). These are **promoted to golden facts** (growing the set beyond 37) at zero fetch cost — they're already anchored and immutable. See Golden Dataset Reference → "Promoted from surplus" for the curated list.

## Testing Strategy

The **6-layer pyramid** (oracle §1), cheapest first. Each layer has a defined remit; if a defect class is *not* in a layer's remit, it is explicitly out of scope.

| Layer | Tool | Location | Catches | Misses |
| --- | --- | --- | --- | --- |
| 0 Schema introspection | pytest + duckdb | `backend/tests/schema/` | Missing views/columns; manifest lineage drift | Wrong cell values |
| 1 SQL invariants | pytest | `backend/tests/invariant/` | Internal inconsistencies (trade double-count, wrong grain) | External disagreement |
| 2 Golden-row fixtures | pytest parametrized over `golden.csv` | `backend/tests/golden/` | End-to-end value correctness for pinned facts | Correctness for unpinned players/seasons |
| 3 Endpoint contract | FastAPI `TestClient` | `backend/tests/contract/` | Response-shape violations, error-code mapping, envelope correctness | Cross-process concerns |
| 4 OpenAPI drift gate | CI workflow | `frontend/scripts/generate-api-types.ts:28-49` | Contract changes (nullability, required-ness, key-set) | Semantic drift inside open-ended types |
| 5 UI smoke + value assertions | Playwright | `frontend/tests/data-correctness/` | Last-mile defects (wrong column rendered, slug 404, `schema_drift` not surfaced) | Anything not asserted on screen |
| 6 Audit reconciliation | SQL against `audit.*` | `backend/tests/audit/` | Known upstream ETL failures before they reach UI | Defects the ETL's own DQ doesn't cover |

### Failure-mode → layer mapping

| Failure mode | Cheapest layer that catches it |
| --- | --- |
| Wrong source object (per-game vs per-season grain) | L0 + manifest lineage (A4) |
| Wrong column, right table | L0 column-semantic manifest (A4) |
| Column-semantic mismatch (season encoding, decimal minutes, pct fraction) | L0 manifest unit/format + L2 golden rows |
| Identifier-space drift (BBR slug 404) | L5 identifier-subset check (A1) |
| OpenAPI/runtime drift | L4 drift gate + runtime `schema_drift` emission |

- **No PBP / shot charts / betting lines in v1** — `fact_pbp_events` (18.7M rows) and `fact_game_betting_lines` are deferred to a Phase 10+. Do not expose them through the generic dataset registry; they need bespoke UI + opponent/team joins.

## Honest Limits (the floor of "for sure")

This plan gets the system to: **"the data on screen is faithful to what the audit-approved ETL produced, the columns are the columns we declared, and ~37 canonical facts reproduce end-to-end."** That is about as close to "for sure" as basketball-data engineering realistically gets.

What this **cannot** catch:
- **Source-of-truth accuracy** — if `dim_player.display_name` is wrong at the source, every test passes. We verify plumbing faithfulness, not source bytes.
- **Coverage ≠ exhaustiveness** — 37 golden rows prove those facts reproduce; they do not prove the next 6,000 players' stats are right. The column-manifest lineage test (A4) is the structural defense; golden rows are the spot defense.
- **Golden-row pinning is only as good as the canonical source** — if Basketball-Reference republishes a historical stat (happens — assists were re-scored in the 1970s), the golden value becomes "wrong but stable" until a human updates `golden.csv`.
- **Runtime-only shape drift inside open-ended types** — `hero_stats: { [key]: unknown }` is a TypeScript escape hatch; zod-parse mitigation covers only consumed keys.

The `audit.*` layer (Decision 5) is the bridge between "plumbing-faithful" and "source-true" — it's the ETL's own DQ verdict, and the UI should always surface `last_dq_status` so the user knows whether they're seeing fresh-passed, fresh-failed, or stale data.

## Track A Results — Discovered Data Divergences (Phase 2 must respect these)

Track A's correctness tests verified every fact against the live 22 GB DuckDB and surfaced **7 divergences** between the DB and the BBR ground truth. None block Phase 2; all affect Phase 2 SQL decisions.

| # | Divergence | Impact | Phase 2 mitigation |
| --- | --- | --- | --- |
| 1 | **Pre-1973 counters stored as `0`, not `NULL`** | `AVG(STL)` across eras is silently wrong (0 dilutes the average). Frontend `PlayerSearchResult.name` maps from `full_name`. |
| 3 | **DuckDB column names are case-sensitive and inconsistent** | Canonical view uses UPPERCASE (`TRB`, `STL`, `BLK`, `FG`, `FT`, `TOV`, `PF`, `G`, `GS`, `MP`); team summary uses lowercase (`w`, `l`); percentages are mixed-case (`3P%`, `2P%`, `eFG%`, `FG%`, `FT%`). | The column manifest (`column_manifest.py`) records the exact lineage for all 66 columns. Phase 2 SQL must match the manifest or the build-time lineage test fails. |
| 4 | **`v_franchise_leaders` includes playoffs** | Kobe = 38,108 (reg+po) not 33,643 (reg only). | Filter `WHERE is_playoffs = false` to match BBR regular-season values. |
| 5 | **`season_year` encoding is mixed** | Some tables use YYYY (ending year), some use YYYY-YY. Career totals need `LIKE '%-%'`; team-season stats use integer YYYY. | Use `api.v_canonical_player_season_totals` (integer `SEASON` = ending year) for canonical queries; fall back to pattern-matching only for career totals. |
| 6 | **Team abbreviations are ambiguous** | Brooklyn stored as `NJN`, Philly as `SYR`; `PHI` maps to 1946-61 Philadelphia Warriors. | Use `team_id` (10-digit NBA ID) in WHERE clauses, not abbreviations. |
| 7 | **ETL has only failed runs** | `audit.pipeline_run_log` has 2 rows, both `status='failed'`. The data exists and is populated but hasn't been verified by a successful ETL pass. | Phase 2 proceeds (views are present + populated). The `audit.*` gate (Decision 5) can report DQ status but cannot yet assert "last run succeeded." UI status-pill must handle the "data present but unverified" state. |
